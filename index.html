Enterprise Structure & Common Setup

Module 1
Why This Comes First: Every Oracle Fusion Financials module depends on Enterprise Structure. No AP invoice can be created without a Business Unit. No GL journal can post without a Primary Ledger. Get this wrong and every downstream module breaks.
🏗️Enterprise Hierarchy
1

Enterprise

Top-level entity — represents your entire organisation. Created once during implementation.

2

Division

Optional grouping between Enterprise and Ledger. Used for large conglomerates.

3

Primary Ledger

Defined by: Chart of Accounts + Calendar + Currency + Accounting Method. All transactions post here.

⚡ Setup first
4

Legal Entity (LE)

Tax reporting entity. Load Geographies → Create Legal Address → Create LE → Assign to PL → Assign BSV.

5

Business Unit (BU)

Operational unit where transactions happen. Create Location → Create BU → Assign PL + Business Functions (Payables, Receivables, Expenses…)

⚡ AP/AR need BU
📐Chart of Accounts (COA) — Key Topics
  • 🔵
    Value Sets — Define the valid values for each segment (Company, Department, Account, Sub-Account, Product)
  • 🔵
    COA Structure + Segments — Define the segment layout; assign Value Sets to each segment; assign Segment Labels (Balancing, Natural Account)
  • 🔵
    Deploy Accounting Key Flexfield (AKF) — Must be deployed before any ledger can be created
  • 🔵
    Segment Values + Value Attributes — Create hierarchy (parent/child), set Allow Posting, Allow Budgeting flags
  • ⚠️
    Best Practices: Only 1 Balancing Segment; BSV = BU; Future Segments reserved for expansion; minimize total segments
Rapid Implementation (RI) Template: Download from Fusion instance → Fill COA, Calendar, Ledger, LE, BU in Excel → Validate → Create Data Files → Upload to create all setups in one shot. Fastest way to configure a greenfield implementation.

Accounts Receivable (AR)

Module 4
AutoAccounting — The Most Critical AR Setup: AutoAccounting rules determine which GL account is debited/credited for every AR transaction (revenue, receivable, tax, freight). If AutoAccounting is wrong, every invoice posts to the wrong GL account.
🧾AR Key Concepts
  • ⚙️
    AutoAccounting — Rule engine mapping transaction types to GL accounts
  • 🏦
    Auto Lockbox — 7-step bank file processing: BAI2 → SFTP → Staging → FBDI → UCM → ESS → LockBox → Apply
  • 💰
    Revenue Recognition — Revenue Scheduling Rules spread recognition over contract periods
  • 📃
    Balance Forward Billing (BFB) — Consolidated billing: multiple transactions on one bill
  • 🔄
    AP/AR Netting — Offset supplier invoices against customer receipts (same legal entity)
  • ✏️
    Receipt Write-Off — Write off small unapplied amounts within approval limits
  • 💸
    Customer Refund — Excess receipt → refund → system creates Payment Request in AP automatically

Q1. What are the four mandatory components of a Primary Ledger in Oracle Fusion?

A
Chart of Accounts, Business Unit, Legal Entity, Currency
B
Chart of Accounts, Accounting Calendar, Currency, Accounting Method
C
COA, Calendar, Currency, Legal Entity
D
COA, Business Unit, Calendar, Currency
A Primary Ledger is defined by exactly 4 pillars: COA + Calendar + Currency + Accounting Method. Legal Entity and BU are assigned after the ledger is created. This is a common trap question.

Q2. In Oracle Fusion AP, what is the correct sequence of the Payment Process Request (PPR)?

A
Build → Select → Confirm → Format
B
Select → Build → Format → Confirm
C
Select → Format → Build → Confirm
D
Build → Format → Select → Confirm
PPR has 4 mandatory stages in order: Select (which invoices) → Build (payment schedule) → Format (payment file) → Confirm (approve and transmit).

Q3. What is AutoAccounting in Oracle Fusion AR and why is it critical?

A
A feature that automatically posts AR invoices to GL without user intervention
B
A rule set that automatically approves AR invoices above a threshold
C
A rule engine that determines the GL accounts (revenue, receivable, tax) for each AR transaction type
D
A batch process that transfers AR balances to GL at period end
AutoAccounting is the core AR configuration that maps transaction types to GL accounts. If wrong, every invoice posts to the wrong revenue and receivable accounts — causing reconciliation failures at period close.

Q4. What is the Mass Additions process in Oracle Fixed Assets?

A
Uploading assets via FBDI template from Oracle OER
B
AP Invoice Lines → Transfer to FA → Prepare Mass Additions → Post Mass Additions
C
Creating asset categories in bulk using ADFdi
D
Running depreciation on all asset books simultaneously
Mass Additions is the standard method to capitalise purchased assets: Create asset lines in AP → Account them → Transfer to FA → Prepare (review/merge/split) → Post. This is the most common integration between AP and FA.

Q5. In Oracle AI Agent Studio, what does RAG (Retrieval-Augmented Generation) enable?

A
Faster processing of LLM responses by caching previous queries
B
Real-time translation of agent responses into multiple languages
C
Grounding agent responses in your specific Oracle data/documents rather than general LLM training
D
Generating visual reports from agent queries automatically
RAG (Retrieval-Augmented Generation) retrieves relevant content from your knowledge base (Oracle docs, SOPs, implementation guides) before generating a response — dramatically improving accuracy for Oracle-specific queries and preventing hallucinations.
